## Введение

  

Процесс — это запущенная программа в операционной системе. Каждый процесс имеет свой идентификатор (PID), использует определенное количество ресурсов (CPU, память) и может взаимодействовать с другими процессами.

  

### Что такое процесс

  

**Процесс** — это экземпляр выполняющейся программы со своим:

  

- Уникальным идентификатором (PID - Process ID)

- Адресным пространством в памяти

- Состоянием (running, sleeping, stopped, zombie)

- Приоритетом выполнения

- Владельцем (пользователем)

- Открытыми файлами и сетевыми соединениями

- Переменными окружения

  

**Пример:**

  

```

Вы запустили браузер Chrome:

- Создается родительский процесс Chrome (PID 1234)

- Создаются дочерние процессы для каждой вкладки

- Каждый процесс использует память и CPU

- Все процессы управляются операционной системой

```

  

### Типы процессов

  

**1. Foreground процессы (Передний план)**

  

```bash

# Процесс блокирует терминал до завершения

./long-running-script.sh

```

  

Вы не можете вводить другие команды, пока процесс не завершится.

  

**2. Background процессы (Фоновые)**

  

```bash

# Процесс работает в фоне, терминал свободен

./long-running-script.sh &

```

  

Терминал свободен для других команд.

  

**3. Daemon процессы (Демоны/Службы)**

  

```

nginx, apache2, mysql, sshd

```

  

Системные службы, работающие постоянно в фоне. Управление службами — в разделе [[#Перезапуск служб]].

  

**4. Zombie процессы (Зомби)**

  

```

Завершенный процесс, но информация о нем еще хранится

Родительский процесс не считал его exit status

```

  

### Состояния процессов

  

| Буква | Состояние | Описание |

|-------|-----------|----------|

| R | Running | Выполняется или готов к выполнению |

| S | Sleeping | Ожидает события |

| D | Disk sleep | Ожидает I/O, uninterruptible |

| T | Stopped | Остановлен сигналом |

| Z | Zombie | Завершен, но не очищен (см. [[#Zombie процессы]]) |

| I | Idle | Бездействующий kernel thread |

  

### Иерархия процессов

  

Все процессы образуют дерево:

  

```

systemd/init (PID 1) — родитель всех процессов

├─ sshd (PID 500)

│ └─ sshd session (PID 1234)

│ └─ bash (PID 1235)

│ └─ vim (PID 1300)

├─ nginx (PID 600)

│ ├─ nginx worker (PID 601)

│ └─ nginx worker (PID 602)

└─ mysql (PID 700)

```

  

- **Parent Process (Родительский)** — процесс, создавший другой процесс

- **Child Process (Дочерний)** — процесс, созданный другим процессом

- **PPID** — Parent Process ID (идентификатор родителя)

## Просмотр списка процессов

  

### Команда ps (Process Status)

  

**Базовое использование:**

  

```bash

# Процессы текущего пользователя в текущем терминале

ps

  

# Вывод:

PID TTY TIME CMD

1234 pts/0 00:00:00 bash

1456 pts/0 00:00:00 ps

```

  

**Показать все процессы:**

  

```bash

# BSD стиль (без дефисов)

ps aux

  

# UNIX стиль (с дефисами)

ps -ef

  

# Оба показывают похожую информацию

```

  

**ps aux — подробное объяснение:**

  

```bash

ps aux

  

# Вывод:

USER PID %CPU %MEM VSZ RSS TTY STAT START TIME COMMAND

root 1 0.0 0.1 169820 12244 ? Ss Oct29 0:03 /sbin/init

www-data 1234 2.5 4.2 450360 86420 ? S 10:00 1:23 /usr/sbin/nginx

mysql 5678 5.0 8.5 1822912 175232 ? Sl Oct29 45:23 /usr/sbin/mysqld

```

  

**Расшифровка колонок:**

  

| Колонка | Описание |

|---------|----------|

| USER | Владелец процесса |

| PID | Process ID (уникальный идентификатор) |

| %CPU | Процент использования CPU |

| %MEM | Процент использования RAM |

| VSZ | Virtual Memory Size (виртуальная память в KB) |

| RSS | Resident Set Size (физическая память в KB) |

| TTY | Терминал (`?` = нет терминала, `pts/0` = псевдотерминал) |

| STAT | Состояние процесса |

| START | Время запуска |

| TIME | Общее время CPU |

| COMMAND | Команда запуска |

  

**Флаги STAT:**

  

| Флаг | Значение |

|------|----------|

| S | Sleeping (спит) |

| R | Running (выполняется) |

| D | Disk sleep (ожидает диск) |

| T | Stopped (остановлен) |

| Z | Zombie (зомби) |

| `<` | Высокий приоритет |

| N | Низкий приоритет |

| L | Заблокированные страницы в памяти |

| s | Session leader |

| l | Многопоточный |

| `+` | В foreground process group |

  

**ps -ef — формат:**

  

```bash

ps -ef

  

# Вывод:

UID PID PPID C STIME TTY TIME CMD

root 1 0 0 Oct29 ? 00:00:03 /sbin/init

root 123 1 0 Oct29 ? 00:00:00 /lib/systemd/systemd-journald

www-data 1234 123 2 10:00 ? 00:01:23 nginx: worker process

```

  

**Расшифровка колонок ps -ef:**

  

| Колонка | Описание |

|---------|----------|

| UID | User ID (числовой) |

| PPID | Parent Process ID (родительский процесс) |

| C | CPU utilization (процент CPU для планировщика) |

| STIME | Start time (время запуска) |

  

**Полезные опции ps:**

  

```bash

# Показать конкретный процесс по PID

ps -p 1234

  

# Показать процессы конкретного пользователя

ps -u username

ps -U root

  

# Показать дерево процессов

ps -ef --forest

ps auxf

  

# Показать потоки (threads)

ps -eLf

  

# Показать с дополнительными колонками

ps aux --sort=-%mem # Сортировка по памяти (убывание)

ps aux --sort=-%cpu # Сортировка по CPU (убывание)

  

# Кастомный формат вывода

ps -eo pid,ppid,user,%cpu,%mem,stat,cmd

```

### Команда top (Table of Processes)

  

**Интерактивный мониторинг процессов в реальном времени:**

  

```bash

top

```

  

**Интерфейс top:**

  

```

top - 10:45:32 up 1 day, 3:21, 2 users, load average: 0.52, 0.58, 0.59

Tasks: 245 total, 1 running, 244 sleeping, 0 stopped, 0 zombie

%Cpu(s): 5.2 us, 2.1 sy, 0.0 ni, 92.5 id, 0.1 wa, 0.0 hi, 0.1 si, 0.0 st

MiB Mem : 16054.2 total, 3421.5 free, 8234.1 used, 4398.6 buff/cache

MiB Swap: 2048.0 total, 2048.0 free, 0.0 used. 6543.2 avail Mem

  

PID USER PR NI VIRT RES SHR S %CPU %MEM TIME+ COMMAND

1234 www-data 20 0 450360 86420 12340 S 5.6 0.5 1:23.45 nginx

5678 mysql 20 0 1822912 175232 23456 S 8.2 1.1 45:23.67 mysqld

9012 user 20 0 789456 45678 8901 S 2.3 0.3 0:12.34 chrome

345 root 20 0 123456 12345 2345 S 0.3 0.1 0:02.45 sshd

```

  

**Расшифровка верхней части top:**

  

| Поле | Описание |

|------|----------|

| `up 1 day, 3:21` | Время работы системы (uptime) |

| `2 users` | Количество залогиненных пользователей |

| `load average: 0.52...` | Средняя нагрузка за 1, 5, 15 минут |

| `Tasks: 245 total` | Всего процессов |

| `1 running` | Запущенных |

| `244 sleeping` | Спящих |

| `0 zombie` | Зомби |

  

**Расшифровка %Cpu:**

  

| Поле | Описание |

|------|----------|

| `5.2 us` | User space (пользовательские процессы) |

| `2.1 sy` | System (kernel) |

| `0.0 ni` | Nice (процессы с измененным приоритетом) |

| `92.5 id` | Idle (простой) |

| `0.1 wa` | Wait (ожидание I/O) |

| `0.0 hi` | Hardware interrupts |

| `0.1 si` | Software interrupts |

| `0.0 st` | Steal time (виртуализация) |

  

**Расшифровка памяти:**

  

| Поле | Описание |

|------|----------|

| total | Всего памяти |

| free | Свободной |

| used | Используется |

| buff/cache | Буферы и кэш |

| avail Mem | Доступно для новых приложений |

  

**Горячие клавиши в top:**

  

| Клавиша | Действие |

|---------|----------|

| `h` / `?` | Справка |

| `q` | Выход |

  

**Сортировка:**

  

| Клавиша | Сортировка |

|---------|------------|

| `P` | По CPU (по умолчанию) |

| `M` | По памяти |

| `T` | По времени работы |

| `N` | По PID |

  

**Фильтрация и поиск:**

  

| Клавиша | Действие |

|---------|----------|

| `u` | Показать процессы конкретного пользователя |

| `k` | Убить процесс (kill) |

| `r` | Изменить nice/priority |

| `L` | Найти строку |

| `&` | Найти следующее |

  

**Отображение:**

  

| Клавиша | Действие |

|---------|----------|

| `1` | Показать все CPU ядра по отдельности |

| `t` | Переключить отображение CPU/tasks |

| `m` | Переключить отображение памяти |

| `c` | Показать полный путь команды |

| `V` | Древовидное отображение (forest view) |

| `i` | Скрыть idle процессы |

  

**Обновление:**

  

| Клавиша | Действие |

|---------|----------|

| `d` / `s` | Изменить интервал обновления (по умолчанию 3 сек) |

| `Space` | Обновить немедленно |

  

**Примеры использования top:**

  

```bash

# Запустить с обновлением каждую 1 секунду

top -d 1

  

# Показать только процессы конкретного пользователя

top -u username

  

# Batch режим (для скриптов, без интерактива)

top -b -n 1

  

# Показать только топ 10 процессов

top -b -n 1 | head -20

  

# Сортировка по памяти при запуске

top -o %MEM

```

  

### Команда htop (улучшенная версия top)

  

**Более удобный и красочный интерфейс:**

  

```bash

# Установка

sudo apt install htop # Ubuntu/Debian

sudo yum install htop # CentOS/RHEL

brew install htop # macOS

  

# Запуск

htop

```

  

**Преимущества htop:**

  

- Цветной интерфейс

- Использование мыши

- Горизонтальная и вертикальная прокрутка

- Визуальное отображение CPU и памяти по ядрам

- Древовидное отображение процессов (tree view)

- Легкое убийство процессов (F9)

- Удобный поиск (F3)

  

**Горячие клавиши htop:**

  

| Клавиша | Действие |

|---------|----------|

| `F1` / `h` | Справка |

| `F2` | Setup (настройки отображения) |

| `F3` / `/` | Поиск процесса |

| `F4` | Фильтр (показать только совпадающие) |

| `F5` | Tree view (древовидное отображение) |

| `F6` | Выбор колонки для сортировки |

| `F7` / `F8` | Увеличить/уменьшить nice (приоритет) |

| `F9` | Убить процесс (kill) |

| `F10` / `q` | Выход |

| `Space` | Пометить процесс |

| `U` | Показать процессы только одного пользователя |

| `t` | Древовидное отображение (tree) |

| `H` | Показать/скрыть user threads |

| `K` | Показать/скрыть kernel threads |

  

### Команда pgrep — поиск процессов

  

Используется вместе с [[#Команда kill]] и [[#Команда pkill — гибкое убийство|pkill]] для управления процессами.

  

**Найти PID процессов по имени:**

  

```bash

# Найти процессы nginx

pgrep nginx

# Вывод:

1234

1235

1236

  

# С полным именем команды

pgrep -f "nginx: worker"

  

# Показать имя процесса вместе с PID

pgrep -l nginx

# Вывод:

1234 nginx

1235 nginx

  

# Показать количество найденных процессов

pgrep -c nginx

# Вывод: 3

  

# Процессы конкретного пользователя

pgrep -u www-data nginx

pgrep -u root

  

# Самый новый процесс с этим именем

pgrep -n nginx

  

# Самый старый процесс

pgrep -o nginx

  

# Инвертировать поиск (все кроме)

pgrep -v bash

```

  

### Команда pidof — найти PID по имени

  

**Проще чем pgrep, но менее гибко:**

  

```bash

# Найти PID процесса nginx

pidof nginx

# Вывод: 1236 1235 1234

  

# Только один PID (самый первый)

pidof -s nginx

# Вывод: 1234

```

## Фильтрация процессов

  

### Фильтрация с помощью grep

  

**Базовая фильтрация:**

  

```bash

# Найти все процессы nginx

ps aux | grep nginx

  

# Вывод:

www-data 1234 0.5 0.3 125728 6392 ? S 10:00 0:01 nginx: master

www-data 1235 0.0 0.2 126100 5234 ? S 10:00 0:00 nginx: worker

root 5678 0.0 0.0 12916 932 pts/0 S+ 10:15 0:00 grep nginx

  

# Исключить строку с самим grep

ps aux | grep nginx | grep -v grep

  

# Или лучше так:

ps aux | grep [n]ginx

```

  

**Расширенная фильтрация:**

  

```bash

# Процессы, использующие больше 1% CPU

ps aux | awk '$3 > 1.0'

  

# Процессы, использующие больше 100MB памяти

ps aux | awk '$6 > 100000'

  

# Процессы конкретного пользователя

ps aux | grep ^www-data

  

# Процессы, запущенные сегодня

ps aux | grep $(date +%H:)

  

# Процессы с определенным состоянием

ps aux | grep " R " # Running

ps aux | grep " Z " # Zombie

  

# Процессы Python

ps aux | grep python

  

# Процессы Node.js

ps aux | grep node

```

  

### Фильтрация по использованию ресурсов

  

**Топ процессов по CPU:**

  

```bash

# Топ 10 процессов по CPU

ps aux --sort=-%cpu | head -11

  

# Топ 5 процессов по CPU (без заголовка)

ps aux --sort=-%cpu | awk 'NR>1' | head -5

  

# Процессы, использующие более 50% CPU

ps aux | awk '$3 > 50'

```

  

**Топ процессов по памяти:**

  

```bash

# Топ 10 процессов по памяти

ps aux --sort=-%mem | head -11

  

# Процессы, использующие более 500MB

ps aux | awk '$6 > 500000' | sort -k6 -rn

  

# Суммарное использование памяти всех процессов пользователя

ps aux | grep ^www-data | awk '{sum+=$6} END {print sum/1024 " MB"}'

```

  

**Процессы по времени работы:**

  

```bash

# Процессы, работающие дольше всех

ps aux --sort=-time | head -11

  

# Процессы, работающие более 1 часа

ps aux | awk '$10 ~ /[0-9]+:[0-9]{2}:[0-9]{2}/'

```

  

### Фильтрация по порту (lsof и netstat)

  

Какой процесс слушает порт — частый вопрос при работе с [[#Перезапуск служб|службами]] (nginx, mysql и т.д.).

  

**Найти процесс, использующий порт:**

  

```bash

# Какой процесс слушает порт 80

sudo lsof -i :80

# Вывод:

COMMAND PID USER FD TYPE DEVICE SIZE/OFF NODE NAME

nginx 1234 www-data 6u IPv4 12345 0t0 TCP *:http (LISTEN)

  

# Какой процесс использует порт 3306 (MySQL)

sudo lsof -i :3306

  

# Все процессы, слушающие TCP порты

sudo lsof -iTCP -sTCP:LISTEN

  

# Процессы с установленными соединениями

sudo lsof -iTCP -sTCP:ESTABLISHED

```

  

**Альтернатива — netstat:**

  

Подробнее о работе с файлами процессов — [[#Фильтрация открытых файлов]].

  

```bash

# Процессы, слушающие порты (с PID)

sudo netstat -tulpn

  

# Расшифровка опций:

# -t: TCP

# -u: UDP

# -l: Listening (слушающие)

# -p: Program (показать PID и имя)

# -n: Numeric (не резолвить имена)

  

# Вывод:

Proto Recv-Q Send-Q Local Address Foreign Address State PID/Program

tcp 0 0 0.0.0.0:80 0.0.0.0:* LISTEN 1234/nginx

tcp 0 0 0.0.0.0:443 0.0.0.0:* LISTEN 1234/nginx

tcp 0 0 127.0.0.1:3306 0.0.0.0:* LISTEN 5678/mysqld

  

# Найти что слушает конкретный порт

sudo netstat -tulpn | grep :80

  

# С ss (современная замена netstat):

sudo ss -tulpn

sudo ss -tulpn | grep :80

```

  

### Фильтрация открытых файлов

  

**Процессы, работающие с файлом:**

  

```bash

# Какой процесс использует файл

lsof /var/log/nginx/access.log

  

# Все файлы, открытые процессом

lsof -p 1234

  

# Все файлы, открытые пользователем

lsof -u www-data

  

# Файлы в конкретной директории

lsof +D /var/www/html

  

# Сетевые соединения процесса

lsof -i -a -p 1234

```

  

### Мониторинг конкретного процесса

  

**Следить за процессом в реальном времени:**

  

```bash

# Мониторинг с помощью watch

watch -n 1 'ps aux | grep nginx'

  

# Постоянный вывод изменений

watch -d 'ps aux | grep [n]ginx'

```

  

**Детальная информация о процессе:**

  

```bash

# Полная информация

ps -p 1234 -f

  

# Расширенная информация

ps -p 1234 -o pid,ppid,user,%cpu,%mem,vsz,rss,stat,start,time,cmd

  

# Переменные окружения процесса

cat /proc/1234/environ | tr '\0' '\n'

  

# Текущая директория процесса

# Статус процесса

```

## Убийство процессов

  

### Сигналы Linux

  

Сигналы — это способ коммуникации между процессами и ядром ОС. Полный список сигналов можно посмотреть командой `kill -l` или `man 7 signal`.

  

**Основные сигналы:**

  

| Номер | Имя | Значение | Можно перехватить |

|-------|-----|----------|-------------------|

| 1 | SIGHUP | Hangup (отключение терминала) | Да |

| 2 | SIGINT | Interrupt (Ctrl+C) | Да |

| 3 | SIGQUIT | Quit (Ctrl+\\) | Да |

| 9 | SIGKILL | Kill (немедленное завершение) | **НЕТ** |

| 15 | SIGTERM | Terminate (вежливое завершение) | Да |

| 18 | SIGCONT | Continue (продолжить выполнение) | Нет |

| 19 | SIGSTOP | Stop (приостановить) | **НЕТ** |

| 20 | SIGTSTP | Terminal stop (Ctrl+Z) | Да |

  

**Важные сигналы подробно:**

  

> [!note] SIGTERM (15) — вежливое завершение

> Процесс получает сигнал и может:

> - Закрыть открытые файлы

> - Освободить ресурсы

> - Сохранить данные

> - Корректно завершиться

>

> Пример: "Пожалуйста, завершись когда будешь готов"

  

> [!warning] SIGKILL (9) — принудительное завершение

> - Процесс **НЕ** может обработать этот сигнал

> - Ядро немедленно завершает процесс

> - Никакой очистки ресурсов

>

> Пример: "Убить немедленно, без вопросов"

  

> [!info] SIGINT (2) — прерывание

> - Обычно вызывается `Ctrl+C`

> - Процесс может обработать и корректно завершиться

> - Используется для остановки foreground процессов

  

> [!tip] SIGHUP (1) — перезагрузка конфигурации

> Часто используется для перечитывания конфигурации без завершения процесса.

>

> Пример: `nginx -s reload`

  

> [!note] SIGSTOP (19) и SIGCONT (18)

> - `SIGSTOP` — приостановить процесс

> - `SIGCONT` — продолжить выполнение

>

> Используется для `Ctrl+Z` (приостановка) и `fg`/`bg` (продолжение)

  

### Команда kill

  

**Базовое использование:**

  

Описание всех сигналов — в разделе [[#Сигналы Linux]].

```bash
# Отправить SIGTERM (15) — вежливое завершение

```

  

**Принудительное завершение:**

```bash
kill -SIGKILL 1234

```

  

**Другие сигналы:**

  

```bash

# Перезагрузить конфигурацию (SIGHUP)

kill -1 1234

  

# Прервать (SIGINT, как Ctrl+C)

kill -INT 1234

kill -2 1234

  

# Приостановить (SIGSTOP)

kill -STOP 1234

kill -19 1234

  

# Продолжить выполнение (SIGCONT)

kill -CONT 1234

kill -18 1234

```

  

**Убить несколько процессов:**

  

```bash

# 1 = процесс не существует или нет прав
```

### Команда timeout — автоматическое завершение

  

Отправляет [[#Сигналы Linux|сигнал]] процессу по истечении времени.

  

**Ограничить время выполнения команды:**

  

```bash

# Убить команду через 10 секунд (SIGTERM)

```

  

### Zombie процессы

  

> [!info] Что такое zombie

> - Процесс завершился, но родитель не считал его exit status

> - Занимает запись в таблице процессов

> - Не использует CPU/память, но отображается в `ps`

> - Состояние `Z` в [[#Состояния процессов]]

> - Родительский процесс — см. [[#Иерархия процессов]]

  

**Найти zombie процессы:**

  

```bash

# Найти все zombie

ps aux | grep Z

  

# Или

ps aux | awk '$8=="Z"'

  

# С pgrep

ps -eo pid,stat,cmd | grep '^[0-9]* Z'

```

  

**Как избавиться от zombie:**

  

```bash

# 1. Найти родительский процесс (PPID)

ps -o ppid= -p 1234 # где 1234 — PID zombie

  

# 2. Убить родителя (он будет перехвачен init/systemd)

kill -HUP <PPID>

  

# 3. Если не помогло — убить родителя принудительно

kill -9 <PPID>

  

# 4. Zombie исчезнет, так как init/systemd считает exit status

```

  

**Автоматическая очистка:**

  

```bash

#!/bin/bash

for pid in $(ps aux | awk '$8=="Z" {print $2}'); do

ppid=$(ps -o ppid= -p $pid)

echo "Killing parent $ppid of zombie $pid"

kill -HUP $ppid

done

```

  

### Безопасное завершение процессов

  

См. также скрипт [[#Автоматическая очистка зависших процессов]] для автоматизации.

  

**Пошаговый подход:**

  

```bash

# 1. Сначала вежливо (SIGTERM)

kill -TERM 1234

  

# 2. Подождать 5 секунд

sleep 5

  

# 3. Проверить, завершился ли

if ps -p 1234 > /dev/null; then

echo "Процесс все еще работает"

# 4. Попробовать SIGINT

kill -INT 1234

sleep 3

# 5. Если не помогло — SIGKILL

if ps -p 1234 > /dev/null; then

echo "Форсируем завершение"

kill -9 1234

fi

fi

```

  

**Скрипт безопасного завершения:**

  

```bash

#!/bin/bash

# safe_kill.sh - безопасное завершение процесса

  

PID=$1

TIMEOUT=10

  

if [ -z "$PID" ]; then

echo "Usage: $0 <PID>"

exit 1

fi

  

# Проверка существования процесса

if ! kill -0 $PID 2>/dev/null; then

echo "Process $PID does not exist"

exit 1

fi

  

# Вежливое завершение

echo "Sending SIGTERM to $PID..."

kill -TERM $PID

  

# Ожидание завершения

for i in $(seq 1 $TIMEOUT); do

if ! kill -0 $PID 2>/dev/null; then

echo "Process $PID terminated successfully"

exit 0

fi

sleep 1

done

  

# Принудительное завершение

echo "Process still running, sending SIGKILL..."

kill -9 $PID

  

if ! kill -0 $PID 2>/dev/null; then

echo "Process $PID killed"

exit 0

else

echo "Failed to kill process $PID"

exit 1

fi

```

  

**Использование:**

  

```bash

chmod +x safe_kill.sh

./safe_kill.sh 1234

```

### Проверка перед перезапуском

**Скрипт безопасного перезапуска:**

  

```bash

#!/bin/bash

# safe_restart.sh

  

SERVICE=$1

  

if [ -z "$SERVICE" ]; then

echo "Usage: $0 <service>"

exit 1

fi

  

# Проверка конфигурации для известных сервисов

case $SERVICE in

nginx)

echo "Checking nginx configuration..."

if ! nginx -t; then

echo "Configuration error! Not restarting."

exit 1

fi

;;

apache2|httpd)

echo "Checking Apache configuration..."

if ! apache2ctl configtest; then

echo "Configuration error! Not restarting."

exit 1

fi

;;

esac

  

# Перезапуск

echo "Restarting $SERVICE..."

if systemctl restart $SERVICE; then

echo "✓ $SERVICE restarted successfully"

systemctl status $SERVICE --no-pager

else

echo "✗ Failed to restart $SERVICE"

exit 1

fi

```

  

### Мониторинг служб

  

Инструменты мониторинга процессов в реальном времени: [[#Команда top (Table of Processes)|top]], [[#Команда htop (улучшенная версия top)|htop]].

  

**Автоматический перезапуск при падении:**

  

systemd может автоматически перезапускать упавшие службы:

  


**Варианты Restart:**

  

| Значение | Описание |

|----------|----------|

| `no` | Не перезапускать (по умолчанию) |

| `on-success` | Перезапускать при чистом выходе |

| `on-failure` | Перезапускать при ошибке |

| `on-abnormal` | Перезапускать при ненормальном завершении |

| `on-abort` | Перезапускать при сигнале abort |

| `always` | Всегда перезапускать |

  

**Мониторинг в реальном времени:**

  

```bash

# Следить за статусом служб

watch -n 1 'systemctl status nginx mysql redis'

  

# Следить за логами нескольких служб

sudo journalctl -u nginx -u mysql -f

  

# Уведомления о падении службы

while true; do

if ! systemctl is-active nginx > /dev/null; then

echo "ALERT: nginx is down!" | mail -s "Nginx Down" admin@example.com

fi

sleep 60

done

```

  

### Восстановление после проблем

  

**Служба не запускается:**

  

```bash

# 1. Проверить статус

sudo systemctl status nginx

  

# 2. Посмотреть полные логи

sudo journalctl -u nginx -n 100 --no-pager

  

# 3. Проверить конфигурацию

sudo nginx -t

  

# 4. Проверить порты (может быть занят)

sudo lsof -i :80

sudo netstat -tulpn | grep :80 # см. [[#Фильтрация по порту (lsof и netstat)]]

  

# 5. Проверить права на файлы

ls -l /var/run/nginx.pid

ls -l /var/log/nginx/

  

# 6. Сбросить failed state

sudo systemctl reset-failed nginx

  

# 7. Попробовать снова

sudo systemctl start nginx

```

  

**Откат после неудачного обновления:**

  

```bash

# Если новая версия не работает

  

# 1. Откатить пакет

sudo apt install nginx=1.18.0-0ubuntu1 # конкретная версия

  

# 2. Или восстановить конфигурацию

sudo cp /etc/nginx/nginx.conf.backup /etc/nginx/nginx.conf

  

# 3. Перезапустить

sudo systemctl restart nginx

```

## Полезные скрипты и примеры

  

Связанные разделы: [[#Просмотр списка процессов]], [[#Убийство процессов]], [[#Перезапуск служб]].

  

### Мониторинг использования ресурсов

  

**Скрипт поиска "прожорливых" процессов:**

  

```bash

#!/bin/bash

# top_processes.sh

  

echo "=== Топ 10 процессов по CPU ==="

ps aux --sort=-%cpu | head -11

  

echo -e "\n=== Топ 10 процессов по памяти ==="

ps aux --sort=-%mem | head -11

  

echo -e "\n=== Процессы, использующие более 50% CPU ==="

ps aux | awk '$3 > 50.0 {print $0}'

  

echo -e "\n=== Процессы, использующие более 500MB памяти ==="

ps aux | awk '$6 > 500000 {print $0}'

```

  

### Автоматическая очистка зависших процессов

  

**Скрипт убийства зависших процессов:**

  

```bash

#!/bin/bash

# kill_hanging.sh

  

PROCESS_NAME=$1

MAX_CPU_TIME=3600 # 1 час в секундах

  

if [ -z "$PROCESS_NAME" ]; then

echo "Usage: $0 <process_name>"

exit 1

fi

  

# Найти процессы, работающие более MAX_CPU_TIME

for pid in $(pgrep $PROCESS_NAME); do

# Получить время CPU в секундах

cpu_time=$(ps -p $pid -o etime= | awk -F: '{print ($1 * 3600) + ($2 * 60) + $3}')

if [ $cpu_time -gt $MAX_CPU_TIME ]; then

echo "Process $pid ($PROCESS_NAME) running for $cpu_time seconds, killing..."

kill -9 $pid

fi

done

```

  

### Групповое управление процессами

  

**Перезапуск группы служб:**

  

```bash

#!/bin/bash

# restart_web_stack.sh

  

SERVICES="nginx php8.2-fpm mysql redis"

  

echo "Restarting web stack..."

  

for service in $SERVICES; do

echo "Restarting $service..."

if systemctl restart $service; then

echo "✓ $service restarted successfully"

else

echo "✗ Failed to restart $service"

exit 1

fi

done

  

echo "All services restarted successfully!"

```

## Заключение

  

Управление процессами — критически важный навык для системного администрирования и DevOps:

  

- **Просмотр процессов** — используйте `ps` ([[#Команда ps (Process Status)|подробнее]]), `top` ([[#Команда top (Table of Processes)|подробнее]]), `htop` ([[#Команда htop (улучшенная версия top)|подробнее]]) для мониторинга. Фильтруйте с помощью `grep`, `pgrep`, `awk` ([[#Фильтрация процессов|подробнее]]).

- **Убийство процессов** — начинайте с вежливого `kill -TERM` ([[#Сигналы Linux|о сигналах]]), используйте `kill -9` только если необходимо. Команды [[#Команда killall — убить по имени|killall]] и [[#Команда pkill — гибкое убийство|pkill]] упрощают убийство по имени.

- **Управление службами** — [[#systemd (современные Linux системы)|systemctl]] для современных систем предоставляет мощный инструментарий: start, stop, restart, reload, enable, disable.

  

> [!tip] Всегда проверяйте конфигурацию перед перезапуском критичных служб и мониторьте логи для диагностики проблем.
